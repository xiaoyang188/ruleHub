# 引言：为什么需要 Skill？

当你第一次用 Claude Code 或 Cursor 写代码时，可能会发现：同一个问题，换一句提问方式，输出质量就天差地别。Agent Skill 正是为了解决这个问题——它把**领域知识、操作步骤和约束条件**打包成一份可被 AI 读取的说明文件，通常命名为 `SKILL.md`。

可以把 Skill 理解成「给 AI 助手的岗位说明书」。普通对话像临时口头交代；Skill 则是经过整理、可版本管理、可分享的标准作业程序（SOP）。

# Skill 解决的核心问题

## 1. 上下文不稳定

大模型每次对话的上下文窗口有限，且无法保证你每次都把背景说全。Skill 在会话开始或任务触发时注入结构化指令，减少「漏说关键约束」导致的返工。

## 2. 经验难以复用

团队里资深工程师的处理套路，往往锁在个人笔记或聊天记录里。把套路写成 Skill 后，新人加载同一 Skill，就能在相近场景下获得一致的行为基线。

## 3. 工具链碎片化

现代 AI 编程涉及 Git、测试框架、部署脚本、Lint 规则等。Skill 可以把「在本仓库里应该怎么跑测试、怎么命名分支」写死，避免 AI 给出与项目规范冲突的建议。

# SKILL.md 通常包含什么？

虽然社区尚未完全统一格式，但高质量 Skill 往往包括：

- **名称与简介**：一句话说明这个 Skill 负责什么
- **适用场景**：什么时候应该启用，什么时候不该用
- **操作步骤**：分步骤的执行清单
- **输入输出约定**：期望用户提供什么，AI 应交付什么
- **反模式与禁忌**：明确不要做什么（例如「禁止直接改生产配置」）
- **示例**：1～2 个完整对话或命令示例

RuleHub 索引的 Skills 大多来自 GitHub 开源仓库。你可以在 [Skills 搜索](/search) 中按关键词、创作者或职业标签筛选。

# Skill 与 Prompt、Rules 的区别

| 概念 | 典型形态 | 主要用途 |
|------|----------|----------|
| Prompt | 单次对话里的自然语言 | 临时任务 |
| Rules | IDE/产品的全局行为约束 | 风格、安全边界 |
| Skill | 独立 Markdown 模块 | 可复用的任务型能力包 |

三者可以叠加：Rules 管底线，Skill 管专项任务，Prompt 管当次微调。

# 如何在日常开发中使用？

## 步骤一：识别重复任务

例如「每次发 PR 前跑 lint + 单测 + 写 changelog 摘要」。若你每周做三次以上，就值得 Skill 化。

## 步骤二：从最小可用版本开始

不必一次写满二十页。先写 10 行：目标、三步流程、两个禁止项。用一周验证后再迭代。

## 步骤三：在团队内共享与版本化

把 `SKILL.md` 放进仓库的 `.cursor/skills/` 或文档目录，走 Git 评审。Skill 和代码一样，需要 Code Review。

## 步骤四：在 RuleHub 发现同类实践

浏览 [创作者目录](/creators) 或 [职业分类](/occupations)，看看其他人是如何把「代码审查」「数据迁移」「文档生成」等场景 Skill 化的，避免从零摸索。

# 常见误区

- **误区一：Skill 越长越好** —— 过长会挤占上下文，重点应清晰、可执行。
- **误区二：Skill 可以替代测试** —— Skill 指导 AI 行为，不能替代自动化测试与人工验收。
- **误区三：复制粘贴就能用** —— 来自其他技术栈的 Skill 需要本地化改造。

# 小结

Agent Skill 是把「会用 AI」升级为「可规模复用 AI 能力」的关键抽象。`SKILL.md` 是载体，真正有价值的是里面沉淀的流程与判断标准。RuleHub 作为 Agent Skills 市场，帮助你发现、比较和学习这些能力模块；而洞察专栏则补充方法论，让你写出的 Skill 更耐用。
